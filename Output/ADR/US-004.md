No ADR required. Three.js is already an established dependency (v0.184.0 in package.json) and the planned `src/render/` module structure aligns with existing repository conventions defined in CLAUDE.md. This ticket implements the rendering layer as originally architected — no new frameworks, no structural departures.

**Architectural notes for implementation:**

1. **Shell resize coordination (medium risk):** `src/ui/shell.ts:19-22` directly sets `canvas.width` and `canvas.height` on resize. Three.js `renderer.setSize()` also writes these properties. The implementer must either (a) remove the shell's resize handler and let the renderer own canvas dimensions, or (b) have the renderer hook into the shell's resize event without calling `setSize()` (only updating camera aspect + projection matrix). Option (a) is cleaner — the shell should return the canvas reference without owning its dimensions post-renderer-bootstrap. This may require a minor refactor of shell.ts.

2. **Test environment mismatch (low risk):** `vitest.config.ts` uses `environment: 'node'`, not jsdom. Pure logic tests (cappedPixelRatio, createScene) work fine in node since THREE.Scene/Camera don't need a DOM. If any test needs `window`, use a per-file `// @vitest-environment jsdom` directive rather than changing the global config.

3. **Playwright WebGL (low risk):** Chromium headless in Playwright supports WebGL by default on macOS. The `--use-gl=angle` flag is only needed in CI Linux environments. The existing playwright.config.ts should work as-is for local development.

4. **Scope is appropriate:** 6 new source files in `src/render/`, 2 test files, and a small modification to `src/main.ts`. No changes to domain or metrics layers. No new dependencies. Estimated complexity of medium is accurate.
