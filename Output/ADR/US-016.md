No ADR required. This ticket adds two new UI modules (`exportPng.ts`, `shareButton.ts`) following the established pattern from `debugHud.ts` and `presetSelector.ts`, plus one pure domain module (`presetUrl.ts`) with no three.js imports. No new dependencies, no new frameworks, no architectural departures.

**Validation notes:**

1. **`preserveDrawingBuffer` confirmed enabled** (`src/render/createRenderer.ts`): The renderer already passes `preserveDrawingBuffer: true` to `WebGLRenderer`, so `canvas.toBlob('image/png')` will capture the current frame correctly. The plan's risk note about this setting being changed is valid — a dev-time assertion or comment would be prudent.

2. **PresetName type confirmed** (`src/domain/presets.ts`): `PresetName = 'fast' | 'mixed' | 'poor'`. The plan's `parsePresetFromUrl` validation against this union is straightforward. Import from `src/domain/presets.ts`, not `types.ts`.

3. **UI module pattern confirmed** (`src/ui/debugHud.ts`, `src/ui/presetSelector.ts`): Both follow the same pattern — create DOM elements, append to overlay, return `{ dispose() }`. The plan correctly mirrors this for export and share buttons. The `data-testid` attributes align with existing Playwright targeting conventions.

4. **main.ts integration point confirmed** (`src/main.ts`): PresetState is created before rendering begins, and `presetState.select()` is the correct method for restoring state from URL. The plan's sequencing (parse URL → select preset → first render) is correct.

5. **vitest environment confirmed** (`vitest.config.ts`): Global environment is `'node'`. The plan correctly recommends per-file `// @vitest-environment jsdom` directives for DOM-dependent tests (exportPng, shareButton) rather than changing the global config. The `presetUrl.test.ts` tests are pure string logic and can run in node.

6. **Clipboard fallback is well-designed**: `navigator.clipboard` requires secure context (HTTPS/localhost). The `window.prompt()` fallback is the standard pattern for non-secure contexts and satisfies the acceptance criterion "No crash if clipboard unavailable."

7. **No shell.ts resize concern**: Unlike the prior US-011 context, `shell.ts` contains no resize handler — it only locates DOM elements. No coordination issue exists for this ticket.

8. **Scope is minimal and additive**: 3 new source files (`src/ui/exportPng.ts`, `src/ui/shareButton.ts`, `src/domain/presetUrl.ts`), 3 test files, 1 E2E spec, and minor wiring in `main.ts`. No changes to render, metrics, or existing domain layers. No new dependencies. Estimated complexity of "medium" is accurate.
