ADR-009 is required and correctly included as step 1 of the plan. The aggregator introduces a new coordination pattern — it is the first component that fuses multiple independent metric sources (ProbeSampler, ResourceTimingCollector) with a mode-switching mechanism (PresetState overrides live) into a single unified output. This is architecturally distinct from both the polling-based probe sampler (ADR-007) and the observer-based resource timing collector (ADR-008). The design decisions that warrant documentation include: (1) cadence strategy — the aggregator runs its own setInterval rather than synchronizing with ProbeSampler's interval, reading latest state on its own tick (a deliberate decoupling choice), (2) preset-override semantics — preset activation short-circuits the entire live aggregation pipeline, which is a mode-switching pattern not present elsewhere, (3) derived metric computation — packetLoss and bandwidth are computed from raw data rather than collected directly, establishing a precedent for derived metrics, and (4) graceful degradation contract — the aggregator guarantees valid numeric output regardless of which sources are present, which is a stronger contract than either individual collector provides alone.

**Architectural alignment:** The plan correctly places the aggregator in `src/metrics/` (not `src/domain/`), which is appropriate since it orchestrates metric collection rather than performing pure normalization. The pure helper functions (`computePacketLoss`, `estimateBandwidth`) being co-located in the aggregator module is acceptable — they depend on metric-layer types (ProbeSample[], ResourceTimingEntry[]) and are unlikely to be reused outside aggregation context.

**Type compatibility verified:** The plan's inputs and outputs align with existing types. `ProbeSampler.getSnapshot()` returns `NetworkSnapshot` (with placeholder packetLoss=0, bandwidth=0). `ResourceTimingCollector.getEntries()` returns `ResourceTimingEntry[]`. `PresetState` exposes `current()` and `subscribe()`. The aggregator's output is `NetworkSnapshot` — the same type already consumed by `mapSnapshotToScene()` in `src/domain/snapshotToScene.ts`. No new domain types are needed.

**Integration risk (low):** Step 10 wires the aggregator into `src/main.ts`, which currently only instantiates PresetState and the renderer. The plan correctly identifies that ProbeSampler and ResourceTimingCollector need to be instantiated here for the first time. The subscription pattern (`subscribe(cb)`) mirrors the existing `PresetState.subscribe()` pattern, maintaining consistency. The aggregator's subscribe callback will feed snapshots to `mapSnapshotToScene()`, preparing the data flow for US-010 without requiring renderer changes now.

**One minor concern:** The plan's `computePacketLoss` uses `rttMs === Infinity` as the failure signal, but `fetchProbe` in `src/metrics/probeSampler.ts` returns `Infinity` for failed probes and `statusCode: 0`. The plan mentions both conditions (Infinity or statusCode 0) — the implementation should use `!isFinite(rttMs)` rather than strict equality to also catch NaN edge cases. This is an implementation detail, not a plan-level issue.

**No other architectural concerns.** The plan adds no new dependencies, stays within established module boundaries, follows existing patterns (start/stop lifecycle, subscribe/unsubscribe, barrel re-exports), and the estimated complexity of "medium" is accurate for ~1 new source file + tests + main.ts wiring.
