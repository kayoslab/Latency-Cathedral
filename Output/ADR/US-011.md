No ADR required. Three.js is already an established dependency (v0.184.0 in package.json) and the planned `src/render/` module structure aligns with existing repository conventions defined in CLAUDE.md. This ticket implements the rendering layer as originally architected — no new frameworks, no structural departures.

**Architectural notes for implementation:**

1. **Shell resize concern is a non-issue:** The pre-plan analysis flagged `src/ui/shell.ts:19-22` as directly setting canvas dimensions, but this is incorrect. Shell.ts (21 lines) only locates DOM elements (`#cathedral` canvas, `#overlay` div) and returns references — it performs no resize handling. Canvas sizing is already fully owned by `initRenderer.ts:45-49` via `renderer.setSize()` + camera aspect update on the `resize` event. No shell refactor is needed.

2. **RendererHandle extension is well-scoped:** The current `RendererHandle` interface (`src/render/initRenderer.ts:7-9`) exposes only `dispose()`. Adding `update(scene: SceneParams)` is the minimal surface needed to wire the aggregator→scene→cathedral pipeline. The `CathedralManager` encapsulates rebuild/dispose lifecycle internally, keeping the public API narrow.

3. **Placeholder mesh removal affects existing tests:** `tests/unit/initRenderer.test.ts:167-175` asserts that `dispose()` calls `renderer.dispose()` and implicitly tests single-mesh disposal (`mesh.geometry.dispose()`, `mesh.material.dispose()`). When the placeholder is replaced by `CathedralManager`, these assertions must be updated to verify the manager's group-traversal disposal instead of single-mesh disposal. The test mock pattern (lines 6-78) will need `Group` added to the three.js mock.

4. **Camera visibility constraint is real:** `createCamera.ts` positions the camera at z=5 with FOV 60°. At the origin, the visible frustum is approximately ±3.5 units wide and ±2 units tall. Cathedral dimensions in `sceneToCathedralParams` must be tuned to fit within this range — a nave height of 2–8 units (as proposed) may clip at the upper end. The implementer should verify visible bounds empirically after initial geometry is rendering.

5. **Rebuild throttling is correctly deferred:** The plan identifies per-tick rebuild as a frame-drop risk but does not include throttling in the implementation steps. This is appropriate — the 2-second aggregator cadence means rebuilds occur at most every 2 seconds, which is well within budget. If future tickets increase aggregation frequency, throttling can be added to `CathedralManager.rebuild()` at that point.

6. **Test environment is correctly handled:** The plan's unit tests should use the `// @vitest-environment jsdom` directive (as `initRenderer.test.ts` does) since `CathedralManager` interacts with three.js Scene objects. Pure domain tests (`cathedralParams.test.ts`) can stay in the default node environment.
