ADR-008 is required and correctly included in the plan. The plan introduces a new telemetry subsystem (`ResourceTimingCollector` using `PerformanceObserver`) that is architecturally distinct from the existing probe sampler — it relies on a different browser API, uses an observer pattern instead of polling, and introduces a new domain type (`ResourceTimingEntry`). These design decisions (observer vs polling, same-origin filtering rationale, rolling window size, graceful degradation strategy) warrant documentation for future contributors. The planned ADR at `docs/adr/008-resource-timing-collection.md` covers the right topics. No other architectural concerns: the plan follows established patterns (types in domain, collector in metrics, re-export from index), adds no new dependencies, and stays within the `src/metrics/` boundary.
